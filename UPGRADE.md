# DataMate 升级文档

本文档用于指导 DataMate 从旧版本升级到新版本。整体流程采用“先备份、再停旧、后部署、再导入”的方式，避免升级过程中旧版本继续写入数据，降低数据不一致风险。

## 升级前准备

升级前需要确认以下信息：

1. 当前部署方式：Docker Compose 或 Kubernetes/Helm。
2. 旧版本和新版本的版本号、镜像地址、部署包来源。
3. 数据库、JWT、对象存储、外部服务等关键配置是否需要沿用。
4. 本地备份目录和可用磁盘空间是否满足数据导出需求。
5. 是否启用了 Milvus、Label Studio、DeerFlow 等可选组件，以及这些组件是否需要同步迁移数据。
6. 是否已安排升级窗口，并停止用户写入或后台任务写入。

## 1. 导出旧版本数据

在旧版本仍可正常访问时，先导出需要迁移的数据。导出范围至少应包含：

| 数据类型 | 说明 |
| --- | --- |
| 数据集文件 | 平台上传、生成或处理后的数据集文件。 |
| 流程文件 | 数据处理流程、编排结果等文件数据。 |
| 数据库数据 | 平台元数据、任务记录、用户配置、业务状态等结构化数据。 |
| 算子文件 | 用户上传或解压后的自定义算子文件。 |
| 运行时依赖 | 自定义算子安装的依赖包或运行时扩展内容。 |
| 日志数据 | 如需保留审计、排障或历史运行记录，可同步导出日志。 |
| 可选组件数据 | Milvus、Label Studio、DeerFlow 等组件启用时，需要按组件分别导出。 |

数据库建议优先使用逻辑备份方式导出，避免直接复制数据库物理目录导致版本兼容或数据一致性问题。文件类数据需要从旧版本容器或旧版本挂载卷中拷贝到本地备份目录。

导出完成后，应检查备份目录是否完整，确认核心数据、数据库备份和可选组件数据都已落盘。

## 2. 将旧版本 Deployment 副本数缩减为 0

数据导出完成后，需要停止旧版本服务，防止旧版本继续写入数据库或文件目录。

使用升级脚本执行新版本部署时，脚本会按以下顺序执行：

1. 在旧版本 Pod 内调用 `export_db.sh` 和 `export_files.sh`，并将 `/migrate_export` 备份到本地。
2. 将旧版本 DataMate Deployment 缩容为 0。
3. 调用 `install.sh` 部署新版本。
4. 在新版本数据库 Pod 内调用 `import_db.sh`，并将备份文件拷贝到新版本文件目标 Pod 的 `/dataset` 目录。

```bash
cd tools
./upgrade.sh -n model-engine --repo <镜像仓库地址> --port 30000 --sc sc-system-manage
```

`upgrade.sh` 通过内置标签识别旧版本资源、迁移 Pod 和新版本 Pod。除升级脚本自身参数外，已支持的安装参数会透传给 `install.sh`。

默认本地备份根目录为 `tools/upgrade-state/<namespace>-backup`。如需指定本地备份根目录，可增加 `--backup-dir`：

```bash
./upgrade.sh -n model-engine --backup-dir /tmp/datamate-upgrade-backup --repo <镜像仓库地址>
```

Kubernetes/Helm 部署场景下，将旧版本相关 Deployment 的副本数缩减为 0。需要处理的服务通常包括：

| 服务 | 处理要求 |
| --- | --- |
| frontend | 停止用户入口。 |
| gateway | 停止 API 网关入口。 |
| backend | 停止 Java 后端写入。 |
| backend-python | 停止 Python 后端写入。 |
| runtime | 停止运行时任务执行。 |
| database | 在确认业务服务停止后再停止数据库。 |
| RayCluster | 如启用，需要暂停任务并停止或缩容计算节点。 |

Docker Compose 部署场景下，停止旧版本容器即可。停止后需要确认旧版本服务不再对外提供访问，也不再执行后台任务。

## 3. 部署新版本

旧版本停止后，使用新版本部署包或新版本镜像部署 DataMate。

部署新版本时需要重点确认：

1. 镜像版本与目标升级版本一致。
2. 命名空间、服务名、端口、域名等访问配置符合预期。
3. 数据库密码、JWT 密钥、外部服务地址等配置与迁移方案一致。
4. 持久化卷配置符合新版本要求。
5. 可选组件是否与旧版本启用范围一致。

新版本部署完成后，先确认所有核心服务处于正常运行状态，再执行数据导入。若新版本会自动初始化数据库或目录结构，应等待初始化完成后再导入旧数据。

## 4. 将导出的数据导入新版本

新版本服务部署完成后，将第一步导出的数据导入新版本环境。

导入顺序建议如下：

1. 导入数据库数据。
2. 导入数据集文件。
3. 导入流程文件。
4. 导入算子文件和运行时依赖。
5. 导入可选组件数据。
6. 根据需要导入历史日志。

导入数据库时，需要确认目标数据库为空库、可覆盖，或已按升级方案完成清理。导入文件时，需要确认目标目录权限、挂载路径和服务用户具备读写权限。

数据导入完成后，重启新版本相关服务，使服务重新加载数据库状态、文件目录和运行时依赖。

## 升级后验证

升级完成后，需要进行功能和数据完整性验证：

1. 前端页面可正常访问。
2. 数据集列表和数据集详情可正常加载。
3. 历史流程可查看，流程文件可访问。
4. 自定义算子可查看、加载和执行。
5. 新建测试任务可正常运行。
6. 后端、Python 后端、运行时、数据库日志无持续异常。
7. 可选组件功能符合预期。

验证通过后，执行确认升级命令清理旧版本 DataMate 资源。该操作会调用 `uninstall.sh` 卸载旧版本 `edatamate` 和 `vdb` release，并删除升级备份目录：

```bash
cd tools
./upgrade.sh -n model-engine --confirm
```

确认完成后，脚本会删除升级备份目录。如果升级时指定了 `--backup-dir`，确认升级时也需要传入同一个目录：

```bash
./upgrade.sh -n model-engine --backup-dir /tmp/datamate-upgrade-backup --confirm
```

## 回滚说明

如果新版本验证失败，应优先停止新版本写入，并根据备份数据回滚。

回滚时需要确认：

1. 旧版本部署包或旧版本镜像仍可用。
2. 旧版本配置仍可恢复。
3. 旧版本数据卷或本地备份仍完整。
4. 新版本运行期间产生的数据是否需要丢弃、合并或单独保留。

如果新版本已经写入过数据，不能直接混用新旧版本数据。应根据升级前导出的备份恢复旧版本环境，避免版本结构差异导致数据异常。

若需要回滚脚本流程，执行：

```bash
cd tools
./upgrade.sh -n model-engine --rollback
```

回滚会调用 `uninstall.sh` 卸载新版本 `datamate` release，并将旧版本 DataMate Deployment 扩容为 1 个副本。
